# DraftCore Agent Guide

本文用于约束后续 agent 在 DraftCore 仓库中的工作方式，目标是让实现持续贴合当前 MVP，而不是边做边扩散。

## 1. 适用范围

- 本文件适用于所有在本仓库内执行分析、设计、编码、测试、文档维护的 agent。
- 若用户口头要求与本文冲突，先解释冲突点，再按用户明确确认后的方向执行。
- 若代码、文档、任务清单三者不一致，不要自行拍板扩展实现，先按“文档优先级”定位依据。

## 2. 项目当前阶段判断

- DraftCore 当前仍处于第一版 MVP 落地阶段，不是平台化建设阶段。
- 目标是跑通“个人、本地、报告优先”的最小闭环，不是一次性覆盖所有文档管理需求。
- 一切实现、重构、补文档、补测试，都应服务于第一版闭环，而不是为未来假设场景预埋复杂机制。

## 3. 不可擅自突破的边界

除非用户明确要求并接受范围变化，否则 agent 不应主动引入以下能力：

- 多人协作、权限、审批、评论流
- 云端同步、在线服务、SaaS 化部署
- OCR、向量检索、复杂召回排序
- Word 导出正式实现
- Excel / PPT 深度处理能力作为第一版承诺
- 与报告主流程无关的通用知识库能力

第一版默认边界：

- 单人使用
- 本地优先
- 报告工作流优先
- 稳定输出以 Markdown 为准
- AI 能力是可选增强，不是基础闭环前提

## 4. 文档优先级

做任何非琐碎改动前，先确认对应约束来自哪里。默认优先级如下：

1. `README.md`：项目目标、阶段定位、总体边界
2. `实施说明.md`：阶段划分、协作方式、范围收缩原则
3. `阶段5任务清单.md`：任务拆分顺序与单次实现粒度
4. `docs/PRD-lite.md`：每个 MVP 任务的输入、输出、成功/失败判定
5. `docs/architecture.md`：技术分层、扩展边界、存储边界
6. `docs/implementation-guide.md`：目录结构、对象草案、接口约定
7. `docs/cli-spec.md` / `docs/schema.md` / `docs/configuration.md`：具体接口与模型约束
8. `docs/testing.md` / `docs/sample-conventions.md`：测试与样例约束

若低优先级文档与高优先级文档冲突：

- 先遵循高优先级文档
- 在提交中补充修正文档，避免冲突长期存在
- 不要把“文档不一致”当作自由发挥的理由

## 5. 工作范式

### 5.1 先收边界，再写代码

- 先识别当前请求属于哪个 MVP 任务或哪个基础模块。
- 先确认该任务的输入、输出、失败条件，再决定实现。
- 不在一次提交里混入多个独立目标；能拆小闭环就拆小闭环。

### 5.2 优先做最小闭环

- 优先完成可演示、可验证、可追溯的最小能力。
- 不为“以后也许要用”提前设计复杂抽象、插件系统或状态机。
- 若后续扩展可以通过新增模块实现，当前不要提前重写既有边界。

### 5.3 改动要与文档保持双向一致

- 新增或修改 CLI 命令时，同时检查 `docs/cli-spec.md` 是否需要同步。
- 修改数据模型、字段、关系时，同时检查 `docs/schema.md`。
- 修改配置来源、默认值、降级行为时，同时检查 `docs/configuration.md`。
- 修改样例结构、测试映射、降级表达时，同时检查 `docs/sample-conventions.md` 与 `docs/testing.md`。

如果改动已改变项目约定，但没有同步文档，这项工作就不算完成。

## 6. 实现约束

### 6.1 目录职责不要混

默认遵循 `draftcore/app/` 下的职责分层：

- `cli/`：只处理命令入口、参数解析、输出编排
- `config/`：配置读取、合并、校验
- `db/`：数据库连接、会话、初始化
- `models/`：数据模型与领域对象
- `services/`：业务编排与流程逻辑
- `parsers/`：不同素材类型的解析实现
- `exporters/`：导出器
- `ai/`：AI Provider 抽象与实现

禁止把业务主逻辑直接堆进 CLI 层，也不要让解析器承担业务流程编排。

### 6.2 扩展优先走边界点

- 新增文件类型支持时，优先新增或扩展 `parsers/`，不要把格式判断散落到业务层。
- 新增输出格式时，优先新增 `exporters/`，不要污染草稿内部模型。
- AI 相关能力必须经过 `ai/` 抽象，不要把第三方模型调用直接写进服务层核心逻辑。

### 6.3 保持降级可用

- 未配置 AI 时，项目、素材、归集、复用、导出、归档应尽量仍可运行。
- 对第一版明确“预留不实现”的能力，应该表现为清晰降级，而不是半实现状态。
- 错误信息要能帮助定位问题，不要用静默失败掩盖边界缺失。

### 6.4 保持可追溯

- 与草稿、复用候选、归档有关的实现，优先保留来源引用和上下文快照。
- 不要为了省事把结构化对象退化成不可追溯的大字符串。
- 输出结果若依赖素材、模板或候选片段，应保留最小来源关系。

## 7. 测试范式

- 所有非纯文案改动，默认都要判断是否需要补测试。
- 单元测试优先验证规则和边界；集成测试优先验证数据库、文件和命令协作。
- 解析器、导出器、复用逻辑的修改，优先补样例驱动测试。
- 自动化测试默认不依赖真实网络和真实模型调用。
- 没有把握补齐端到端时，至少补到能阻止回归的最小层级。

推荐最先保持可运行的命令：

```powershell
pytest
pytest tests/unit
pytest tests/integration
```

## 8. 提交质量要求

- 每次提交只解决一个清晰问题，避免“顺手重构一大片”。
- 新增代码应优先可读、可维护，而不是炫技巧。
- 不擅自改动无关文件，不顺手重排整仓风格。
- 若发现用户工作区已有未说明改动，默认视为他人工作成果，不覆盖、不回退。

## 9. 何时需要先停下来确认

遇到以下情况，应先说明风险再继续，而不是自行猜测：

- 请求明显突破当前 MVP 边界
- 多份核心文档之间存在实质冲突
- 需要变更 CLI 公开接口、核心模型字段或归档语义
- 需要引入新依赖，且其影响超出当前任务
- 需要修改大量现有结构才能完成一个本应很小的需求

## 10. 对后续 agent 的默认期待

后续 agent 应默认做到：

- 先读相关文档，再动实现
- 先完成最小闭环，再考虑扩展
- 先补可解释测试，再宣布完成
- 先维护边界一致性，再追求功能数量

如果拿不准，就优先做更小、更稳、更贴近文档的一版。
