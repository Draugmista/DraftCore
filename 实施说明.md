# 个人开源项目实施说明

## 适用范围

本说明用于规划一个辅助本地报告、PPT 和 Excel 编写的个人开源项目。项目聚焦以下两类能力：

- 素材管理
- 成果管理

项目采用精简流程推进，由你个人与 Codex 协作完成。本文只说明各阶段要做什么，以及每个阶段的输入与输出，不涉及技术选型与实现细节。

## 阶段 0：项目定位与协作约定

### 干什么

明确项目目标、使用边界、协作方式和整体节奏，避免一开始就做大做散。

### 输入

- 你当前在报告、PPT、Excel 编写中的真实痛点
- 你对“素材管理”和“成果管理”的基本预期
- 个人项目的时间投入和维护预期

### 输出

- 项目目标说明：本项目优先解决你个人在本地报告编写中的整理、查找、复用、改写效率问题。项目长期背景来自报告、PPT 和 Excel 编写需求，但阶段 0 明确第一版以报告类成果为中心，不追求一开始覆盖全部写作形态。
- 项目边界说明：本项目当前只面向个人使用、本地文件和轻量流程。首批素材范围默认覆盖本地常见文件，包括文档、图片、表格、历史成果、模板、参考资料。MVP 仅验证报告链路，PPT 和 Excel 保留为后续扩展方向，不纳入当前第一版交付承诺。本阶段不引入多人协作、云端同步和大而全的平台化能力。
- 协作约定说明：你负责项目目标、关键取舍、结果确认和最终验收；Codex 负责辅助分析、结构化整理、方案拆解、内容起草、任务分解和实现支持。阶段 0 只形成项目定位、项目边界和协作约定，不提前扩展为技术选型、架构设计或完整功能清单。

### 完成标准

- 形成一页式阶段 0 共识稿，可直接作为阶段 1 到阶段 5 的统一输入
- 读完后能够明确说明项目当前主要解决的问题
- 读完后能够区分当前阶段做什么，以及明确不做什么
- 后续若讨论偏向 PPT、Excel、多人协作、云端同步，可明确判定为超出当前 MVP
- 进入阶段 1 时，不需要再次讨论你与 Codex 的职责分工

## 阶段 1：使用场景梳理

### 干什么

围绕“个人、本地、报告优先”的前提，梳理项目实际服务的工作场景，优先聚焦高频、重复、容易混乱的个人工作流程，只保留会直接影响 MVP 的场景。

### 输入

- 你过往编写本地报告、PPT、Excel 的实际过程
- 常见输入材料，如文档、图片、表格、历史成果、模板、参考资料
- 常见输出成果，如汇报稿、分析表、演示文稿、阶段性文档

### 输出

- 典型使用场景清单：
  1. 从历史报告、模板和参考材料中查找可复用内容，用于新报告起草。
  2. 把零散素材整理成可引用的报告输入，包括文档、表格、图片和历史成果。
  3. 围绕某个主题或任务，快速归集相关素材，避免素材分散和遗漏。
  4. 基于已有素材生成报告初稿结构，形成章节、要点或内容骨架。
  5. 在报告编写过程中反复改写、精简、补充和统一表达，减少重复劳动。
  6. 在报告完成后归档成果，并保留与所用素材的对应关系，方便后续复用。
- 场景优先级排序：
  1. 历史内容与模板复用
  2. 零散素材归集与整理
  3. 报告初稿结构生成
  4. 编写过程中的改写与统一
  5. 成果归档与回溯
- 首批要解决的问题列表：
  1. 报告素材分散在不同文件和目录里，临时查找成本高。
  2. 历史报告和模板虽然可用，但难以快速定位到可复用段落、结构或数据。
  3. 同一主题下的文档、图片、表格、参考资料之间缺少清晰归集关系。
  4. 每次起草新报告时，常常要从头整理结构，重复劳动较多。
  5. 写作过程中改写、合并、统一口径比较耗时。
  6. 报告完成后，成果与素材之间缺少稳定对应，后续复用又要重新找。

### 完成标准

- 形成一版围绕个人本地报告工作的典型场景清单
- 完成场景优先级排序，能够支持后续 MVP 范围收缩
- 明确首批要解决的问题，且这些问题都能追溯到具体场景
- 阶段 2 开始时，不需要再次讨论“当前 MVP 主要服务哪些报告工作环节”

## 阶段 2：素材与成果对象定义

### 干什么

明确当前 MVP 真正要管理的对象，只保留支撑本地报告编写所需的最小信息，不提前引入复杂标签体系、多人协作信息或平台级数据模型。

### 输入

- 阶段 1 的场景清单
- 你常用的素材类型和成果类型
- 你当前已有的目录习惯、命名习惯、归档习惯

### 输出

- 素材对象清单：
  1. 原始素材：适用于单个输入文件或资料项，如文档、表格、图片、参考资料、历史报告片段，是最基础的可收集、可归档、可引用对象。
  2. 素材集合：适用于围绕某个报告主题或任务聚合的一组素材，用于把分散文件组织成可使用的输入上下文。
  3. 模板素材：适用于报告模板、固定章节框架、常用表达、常用结构样式，主要承担复用骨架作用。
  4. 参考成果：适用于历史报告、阶段性文档、已完成成果，既是素材来源，也是新报告可复用的参考对象。
- 成果对象清单：
  1. 报告项目：表示一次具体报告任务，是当前 MVP 的核心成果容器，用于串联主题、目标、相关素材和产出内容。
  2. 报告草稿：表示编写过程中的阶段性内容，用于起草、改写和调整结构，强调过程态。
  3. 最终报告：表示确认完成并归档的成果版本，强调结果态和后续复用价值。
- 对象之间的关系说明：
  1. 一个报告项目可关联多个原始素材、模板素材和参考成果。
  2. 多个原始素材可归入一个素材集合，一个素材集合可服务于一个或多个报告项目。
  3. 模板素材为报告草稿提供结构、章节框架或表达模板。
  4. 参考成果既可作为新项目输入，也可在项目完成后沉淀为新的参考成果。
  5. 一个报告项目可包含多个报告草稿，但当前 MVP 默认只强调一个主草稿持续演进。
  6. 报告草稿验收完成后形成最终报告，并与实际使用过的素材保持可追溯关系。
- 最小管理信息清单：
  1. 原始素材：名称、文件路径、文件类型、来源类别、所属主题或任务、简要说明、是否已被当前报告使用。
  2. 素材集合：集合名称、对应主题或任务、包含的素材项、用途说明。
  3. 模板素材：模板名称、模板类型、适用场景、文件路径、备注。
  4. 参考成果：成果名称、成果类型、对应主题、文件路径、可复用说明。
  5. 报告项目：项目名称、报告主题、当前状态、关联素材、目标输出。
  6. 报告草稿：草稿名称、所属项目、版本标识、更新时间、当前用途。
  7. 最终报告：成果名称、所属项目、定稿时间、文件路径、对应素材范围。

### 完成标准

- 形成一版围绕报告 MVP 的最小对象清单
- 明确区分素材对象与成果对象，避免后续流程拆解时概念混用
- 明确对象之间的基本关系，能够支持阶段 3 的流程闭环设计
- 明确每类对象的最小管理信息，且这些信息足以支持查找、复用、归集和归档

## 阶段 3：核心流程拆解

### 干什么

把日常工作拆成最小闭环，明确从收集素材到产出成果的关键步骤，压缩不必要流程，只保留支撑个人、本地、报告优先场景的关键动作。

### 输入

- 阶段 2 的对象定义
- 你当前实际操作步骤
- 常见卡点、重复动作和容易遗漏的环节

### 输出

- 素材管理流程：
  1. 收集素材：收集当前报告相关的原始素材、模板素材和参考成果。
  2. 归入主题：按当前报告主题或任务将素材归入素材集合，避免继续分散。
  3. 标记用途：为素材补充最小必要用途说明，如背景参考、数据来源、结构模板、历史复用内容。
  4. 筛出可用项：从素材集合中筛出当前报告实际使用的输入，区分已收集素材和实际使用素材。
- 成果管理流程：
  1. 建立报告项目：围绕当前任务建立报告项目，明确报告主题、目标输出和关联素材范围。
  2. 生成主草稿：基于模板素材、参考成果和已确认素材，形成报告主草稿的结构和初稿内容。
  3. 持续改写完善：在主草稿上持续补充、改写、合并和统一表达，逐步形成可验收版本。
  4. 确认并定稿：当报告内容满足当前任务需要后，将主草稿转为最终报告。
  5. 归档并沉淀：归档最终报告，同时保留其与素材、模板、参考成果的关联关系，供后续复用。
- 从素材到成果的最小闭环流程：
  1. 建立报告项目。
  2. 收集并归集相关原始素材、模板素材和参考成果。
  3. 从素材集合中筛出当前实际使用的输入。
  4. 在素材足以支撑当前章节时边收边写，生成主草稿结构和初稿内容。
  5. 持续修订主草稿，直至满足当前任务要求。
  6. 输出最终报告，并将成果沉淀为后续可复用的参考成果。
- 需要保留、简化、取消的流程项：
  1. 保留：围绕具体报告任务归集素材；区分原始素材、模板素材和参考成果；采用一个主草稿持续演进；保留最终报告与所用素材的可追溯关系。
  2. 简化：素材录入只记录最小必要信息；素材状态只区分已收集和实际使用；草稿版本只保留必要节点，不建立复杂版本体系。
  3. 取消：先全量整理完所有素材再开始写报告；在 MVP 阶段引入多人评审流、审批流、共享流；建立复杂命名规范、表单登记流程和大而全知识库。

### 完成标准

- 形成一版围绕报告 MVP 的素材管理流程和成果管理流程
- 明确从素材到最终报告的最小闭环，能够支持边收边写的实际工作方式
- 明确保留、简化、取消的流程项，能够为阶段 4 的 MVP 裁剪提供依据
- 阶段 4 开始时，不需要再次讨论是否采用多草稿流、重型归档流或全量整理流

## 阶段 4：MVP 范围确定

### 干什么

确定第一版只做什么、不做什么，优先覆盖最有价值的个人使用路径，以管理优先、闭环优先的方式跑通报告 MVP。

### 输入

- 阶段 3 的流程拆解结果
- 你当前最迫切需要解决的问题
- 个人项目可接受的开发与维护成本

### 输出

- MVP 功能范围：
  1. 建立报告项目并关联当前任务素材。
  2. 收集、归集、标记和筛选原始素材、模板素材、参考成果。
  3. 支持从历史内容和模板中定位可复用内容。
  4. 基于已确认素材生成报告主草稿骨架或初稿。
  5. 在主草稿上做最小改写、补充和统一表达。
  6. 输出最终报告并完成基础归档。
  7. 保留最终报告与所用素材、模板、参考成果的基础可追溯关系。
- 暂不纳入的功能清单：
  1. PPT、Excel 的独立流程与能力。
  2. 多人协作、共享、审批、评审流。
  3. 云端同步与平台化能力。
  4. 复杂标签体系、复杂版本体系、完整知识库建设。
  5. 重型归档卡片、复杂统计分析、自动化规则编排。
  6. 高级内容生成能力，如大篇幅自动写作、复杂结构规划、深度智能推荐。
- MVP 验收标准：
  1. 能围绕一个真实报告任务建立报告项目。
  2. 能把相关素材归集成可使用输入，并区分已收集与实际使用。
  3. 能从模板或历史成果中提取并复用结构或内容。
  4. 能生成并持续修订一个主草稿，直至形成最终报告。
  5. 能在报告完成后保留其与素材的基础关联，支持后续复用。
  6. 全流程不依赖多人协作、审批或复杂人工登记，单人可独立完成闭环。

### 完成标准

- 形成一版管理优先、闭环优先的第一版 MVP 范围说明
- 明确区分第一版要做与明确后置的能力边界
- 第一版范围完整覆盖素材归集、报告起草、草稿修订和最终归档的最小闭环
- 阶段 5 开始时，不需要再次讨论是否纳入多人协作、复杂版本流、重型归档或 PPT、Excel 流程

## 阶段 5：实施任务拆分

### 干什么

把 MVP 拆成可逐步完成的小任务，形成你与 Codex 可以持续协作推进的任务序列，按闭环小步推进，每个任务都能交付一个可演示、可验证的小能力。

### 输入

- MVP 功能范围
- 你希望的迭代节奏
- 你希望优先验证的功能顺序

### 输出

- 阶段性任务列表：
  1. 报告项目与素材收集任务：能建立报告项目，并把当前任务相关的原始素材、模板素材、参考成果纳入管理范围。
  2. 素材归集与用途标记任务：能把分散素材归入素材集合，并补充最小必要用途说明。
  3. 复用定位任务：能从历史报告和模板中定位可复用结构或内容，并关联到当前报告项目。
  4. 主草稿生成任务：能基于已确认素材生成报告主草稿骨架或初稿。
  5. 主草稿改写任务：能在主草稿上做补充、改写、合并和统一表达。
  6. 最终报告归档任务：能输出最终报告，并保留与素材、模板、参考成果的基础关联。
- 每个任务的目标说明：
  1. 只解决一段明确使用路径中的一个关键问题。
  2. 完成后能支撑下一任务继续推进。
  3. 不提前带入后续复杂能力。
- 每个任务的前置条件与完成标志：
  1. 前置条件：上一个闭环任务已可演示，且输入对象已明确。
  2. 完成标志：能用一组真实样例完成一次演示，看到明确输入、处理和输出；结果可被你确认是否满足当前预期。

### 完成标准

- 形成一版按闭环小步推进的阶段性任务序列
- 明确每个任务的目标说明，并保证任务之间有清晰承接关系
- 明确每个任务的前置条件与完成标志，能够支持阶段 6 按顺序实施和试运行
- 阶段 6 开始时，不需要再次讨论先做素材管理还是先做起草生成，也不需要再次讨论任务完成到什么程度算通过

## 阶段 6：MVP 开发与试运行

### 干什么

按任务顺序逐步实现，并用真实资料和真实成果进行试运行，验证是否真正提升效率。

### 输入

- 阶段 5 的任务列表
- 真实的素材样本
- 真实的报告、PPT、Excel 编写需求

### 输出

- 可用的 MVP 版本
- 试运行记录
- 问题清单与优化建议

## 阶段 7：个人使用优化

### 干什么

根据试运行情况，优先修正影响日常使用的问题，提升稳定性、清晰度和可持续使用体验。

### 输入

- 试运行记录
- 问题清单
- 你在真实使用中的反馈

### 输出

- 优化后的可持续使用版本
- 已解决问题清单
- 后续改进候选项

## 阶段 8：开源整理与持续迭代

### 干什么

把项目整理成适合公开维护的形式，同时保持个人项目所需的轻量节奏。

### 输入

- 已稳定的项目版本
- 项目目标、使用方式和边界说明
- 后续改进候选项

### 输出

- 对外可理解的项目说明
- 面向后续迭代的待办清单
- 适合个人长期维护的迭代节奏

## 推荐推进顺序

建议按以下顺序推进：

1. 先把场景和边界说清楚
2. 再把素材、成果和流程讲清楚
3. 然后收缩出 MVP
4. 最后进入小步快跑的实现和迭代

这样可以保证项目从一开始就围绕真实个人需求，而不是围绕工具本身扩张。
