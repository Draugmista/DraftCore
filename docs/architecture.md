# DraftCore 总体架构

## 1. 文档目的

本文用于承接 [实施说明.md](../实施说明.md) 与 [阶段5任务清单.md](../阶段5任务清单.md)，把阶段 6 进入实现前必须固定的总体技术架构说明清楚。

本文只覆盖第一版 MVP 的技术架构，不重复产品范围讨论。默认前提如下：

- 单机、本地优先、个人使用
- 报告工作流优先
- 第一版产品主输出为 Markdown
- 架构必须兼容后续 Word 输出扩展
- 素材对象从一开始覆盖图片、Markdown、Word、PPT 等多类型本地文件

## 2. 系统定位

DraftCore 是一个面向个人报告工作流的本地 CLI 工具，用于把分散在本地目录中的多类型素材组织成可复用输入，并支撑从素材归集、复用定位、主草稿生成、改写到最终归档的最小闭环。

第一版的系统目标不是做成通用知识库，也不是做高保真文档编辑器，而是：

- 让多类型本地素材进入统一管理范围
- 让素材与报告项目、主草稿、最终报告之间建立可追溯关系
- 让 Markdown 成为第一版稳定输出格式
- 为后续 Word 工作流保留解析与导出扩展点

## 3. 核心架构原则

### 3.1 本地文件优先

- 原始素材、模板、参考成果、草稿与最终输出文件均保存在本地文件系统
- 系统不复制原始大文件，不建立云端文件仓库
- 数据库存储的是路径、元数据、抽取结果摘要和对象关系

### 3.2 文件类型适配优先于业务耦合

- 业务层不直接处理 `md`、`docx`、`pptx`、图片等格式细节
- 文件解析通过统一适配接口接入
- 业务模块只依赖标准化后的素材信息

### 3.3 Markdown 先落地，Word 先预留

- 第一版草稿和最终报告默认导出为 Markdown
- 草稿内部模型不直接等同于 Markdown 字符串
- 导出层负责把结构化草稿渲染为 Markdown
- 后续增加 Word 导出时，不应要求重写项目、素材、复用、草稿核心逻辑

### 3.4 AI 为增强能力，不是基础闭环前置条件

- 没有 AI Provider 配置时，项目、素材、归集、检索、归档仍应可用
- AI 优先用于主草稿生成和改写增强
- AI 输入应基于标准化素材片段，而不是绑定某一原始文件格式

## 4. 支持的素材范围

第一版架构中，素材对象统一归为 `Asset`，按文件类型进行处理。

| 类别 | 典型格式 | 第一版直接支持 | 后续扩展方向 |
| --- | --- | --- | --- |
| 文本类 | `md` `txt` | 读取全文、提取摘要、参与检索与复用 | 更细粒度分段和结构识别 |
| 文档类 | `docx` | 提取段落文本、标题层级、基础元数据、参与检索与复用 | 更完整的样式语义、批注或修订信息 |
| 演示类 | `pptx` | 提取页标题、文本框文本、备注等最小文本信息 | 更深层版式理解、图表结构识别 |
| 图像类 | `png` `jpg` `jpeg` 等 | 管理路径、名称、备注、用途、项目关联 | OCR、图像内容理解 |
| 表格类 | `xlsx` | 第一版仅预留类型和接口，不作为 MVP 必做能力 | 表格文本抽取、结构识别 |
| 参考成果类 | 历史报告、模板、阶段文档 | 以其实际文件格式进入系统，再按来源类别区分用途 | 更完整的成果复用策略 |

说明：

- “素材类型”与“来源类别”是两套概念
- 素材类型描述文件格式与解析方式
- 来源类别描述它在业务中的角色，例如原始素材、模板素材、参考成果

## 5. 模块划分

### 5.1 `project`

负责报告项目管理，包括：

- 创建、列出、查看报告项目
- 维护项目主题、目标输出、状态
- 维护项目与素材、草稿、最终报告的关联

### 5.2 `asset`

负责多类型素材登记与元数据管理，包括：

- 新增素材记录
- 维护文件路径、文件类型、来源类别、用途说明
- 维护素材是否已被当前项目使用
- 维护素材抽取状态

### 5.3 `ingestion/parsers`

负责按文件类型做解析适配，包括：

- 根据扩展名或 MIME 类型选择解析器
- 读取基础元数据
- 抽取最小可检索文本或结构片段
- 输出统一标准化结果

该层是多格式扩展的关键边界。新增格式支持时，优先新增解析器，不修改业务层。

### 5.4 `collection`

负责围绕当前项目组织素材集合，包括：

- 按主题或任务建立素材集合
- 管理用途标记
- 输出项目当前可用输入范围

### 5.5 `reuse`

负责复用定位，包括：

- 基于关键词、文件类型、来源类别、用途说明进行检索
- 对支持文本抽取的素材利用抽取文本进行内容定位
- 产出可复用结构、片段或参考来源清单

第一版不引入向量检索，不依赖 embedding 或独立检索服务。

### 5.6 `draft`

负责主草稿生成与持续改写，包括：

- 建立项目主草稿
- 基于标准化素材片段与复用结果生成结构化草稿
- 持续改写、补充、合并和统一表达

草稿内部应采用“结构化章节 + 文本块”模型，而不是直接把业务状态绑定到 Markdown 语法。

### 5.7 `export`

负责输出渲染，包括：

- 第一版实现 `MarkdownExporter`
- 接收结构化草稿和元数据
- 输出 Markdown 文件

此层必须预留 `WordExporter` 扩展位，后续把同一份结构化草稿渲染为 `docx`。

### 5.8 `archive`

负责最终成果归档，包括：

- 主草稿转最终报告
- 保存最终输出路径
- 记录与素材、模板、参考成果的基础追溯关系

### 5.9 `ai_provider`

负责 AI 能力接入，包括：

- 统一封装模型调用
- 提供主草稿生成和改写接口
- 屏蔽具体云端供应商差异

业务层只调用能力接口，不直接耦合具体模型 SDK。

## 6. 数据流

### 6.1 素材进入系统

1. 用户创建报告项目
2. 用户登记本地素材文件
3. `asset` 记录基础元数据
4. `ingestion/parsers` 按文件类型抽取标准化内容
5. 系统把素材关联到项目和素材集合

### 6.2 复用定位

1. 用户在项目上下文中执行复用检索
2. `reuse` 基于元数据和已抽取文本筛选候选
3. 系统输出可复用结构、历史片段、模板参考
4. 结果可回写为项目的复用候选记录

### 6.3 草稿生成与导出

1. `draft` 读取项目、素材集合与复用候选
2. 如启用 AI，则通过 `ai_provider` 生成章节草稿或改写结果
3. 形成结构化草稿模型
4. `export` 把草稿渲染为 Markdown 文件
5. `archive` 在定稿后记录最终报告与素材追溯关系

## 7. 存储边界

## 7.1 SQLite 保存内容

SQLite 用于保存以下信息：

- 报告项目
- 素材对象
- 素材抽取摘要和索引字段
- 素材集合与用途标记
- 复用候选
- 草稿元数据与结构化内容
- 最终报告记录
- 对象之间的关联关系

## 7.2 文件系统保存内容

文件系统保存以下内容：

- 原始素材文件
- 模板和历史成果文件
- Markdown 草稿和最终输出文件
- 后续扩展的 Word 导出文件

## 7.3 不做的事情

第一版不做以下设计：

- 不复制原始素材正文到数据库
- 不建立独立对象存储
- 不引入向量数据库
- 不引入常驻后台服务

## 8. 扩展边界

### 8.1 Word 工作流扩展

后续扩展 Word 时，只允许在以下位置新增能力：

- `ingestion/parsers` 中增强 `docx` 解析
- `export` 中新增 `WordExporter`
- 如有必要，在草稿结构模型中补充样式语义字段

不应因 Word 扩展而重写以下模块：

- `project`
- `asset`
- `collection`
- `reuse`
- `archive`

### 8.2 语义检索扩展

如未来需要语义检索，应在 `reuse` 模块下增加独立检索适配层，不直接污染素材登记和草稿模型。

### 8.3 图像 OCR 扩展

如未来需要图片 OCR，应作为图像解析器增强项接入，不改变 `Asset` 的基础对象定义。

## 9. 当前实现边界与后置边界

### 9.1 第一版直接实现

- 多类型素材登记
- `md`、`txt`、`docx`、`pptx` 的最小文本/结构抽取
- 图片素材的路径级管理
- 基于关键词和元数据的复用定位
- 结构化草稿模型
- Markdown 导出
- 最终报告归档与追溯

### 9.2 第一版只预留，不实现

- Word 导出
- 图片 OCR
- 向量检索
- PPT 深度结构理解
- 复杂版式高保真还原

## 10. 架构验收标准

当以下条件成立时，可认为总体架构已足以指导实现：

- 开发者能明确每个业务能力对应哪个模块
- 开发者能区分素材类型、来源类别和输出格式
- 开发者能说明为何第一版输出选 Markdown、同时如何兼容后续 Word
- 新增 `WordExporter` 时，不需要重做项目、素材、复用和归档逻辑
- 当前架构能完整支撑阶段 5 的六个任务闭环
