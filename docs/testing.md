# DraftCore 测试策略

## 1. 目的

本文定义 DraftCore 第一版的测试边界、分层方式和最小验收策略，确保项目在开始编码后有明确的质量基线，而不是等功能堆起来后再补测试。

## 2. 测试原则

- 优先覆盖 MVP 主流程
- 优先验证可追溯和可落地，而不是追求高覆盖率数字
- 测试要能解释失败原因
- 多格式解析器必须有样例驱动测试
- AI 能力默认通过接口隔离，避免测试依赖真实云端调用

## 3. 测试分层

## 3.1 单元测试

覆盖范围：

- 路径处理
- 配置加载与校验
- 数据模型校验
- 解析器最小行为
- 导出器渲染逻辑
- 关键词检索与复用候选生成规则

目标：

- 快速
- 稳定
- 不依赖外部网络

## 3.2 集成测试

覆盖范围：

- SQLite 初始化与数据读写
- 项目、素材、集合、复用、草稿、导出、归档的跨模块协作
- CLI 命令触发后对数据库和文件系统的实际影响

目标：

- 验证模块拼接后的行为正确
- 验证关系和状态能真实落盘

## 3.3 端到端验收测试

覆盖范围：

- 对照 PRD 的六个任务闭环
- 使用真实样例素材跑通最小工作流

目标：

- 验证“不是代码看起来对”，而是用户场景真的能走通

## 4. 样例驱动测试要求

解析器和工作流测试都应尽量基于固定样例素材。

建议准备：

- `samples/assets/`
- `samples/expected/`
- 命名和映射规则见 [样例约定](sample-conventions.md)

样例内容至少包含：

- 正常文本素材
- 可部分抽取的文档素材
- 仅路径级管理的图片素材
- 模板素材
- 历史参考成果

每个样例最好都能回答：

- 输入是什么
- 期望抽取什么
- 哪些字段允许为空
- 哪些失败属于正常降级

## 5. 各模块最小测试要求

## 5.1 配置模块

至少覆盖：

- 读取默认配置
- 环境变量覆盖
- AI 配置缺失时的降级
- 非法路径或非法字段时报错

## 5.2 数据库与模型

至少覆盖：

- 表初始化成功
- 必填字段校验
- 枚举状态合法性
- 关系表写入和查询

## 5.3 解析器

### `md` / `txt`

至少覆盖：

- 读取全文
- 生成摘要
- 可检索文本非空

### `docx`

至少覆盖：

- 段落抽取
- 标题层级或近似结构抽取
- 部分抽取时的降级状态

### `pptx`

至少覆盖：

- 页标题抽取
- 文本框文本抽取
- 备注区缺失时不应整体失败

### 图片

至少覆盖：

- 路径与基础元数据登记
- 不做 OCR 时仍可形成合法结果

## 5.4 复用定位

至少覆盖：

- 模板素材筛选
- 参考成果筛选
- 关键词命中生成候选
- 只有路径级参考时的降级表达

## 5.5 草稿与导出

至少覆盖：

- 结构化草稿生成
- Markdown 导出
- 导出路径写入正确

## 5.6 归档

至少覆盖：

- 最终报告记录创建
- 输出文件存在
- 报告与素材/复用候选关系可追溯

## 6. AI 相关测试策略

第一版不建议在自动化测试中依赖真实模型调用。

建议做法：

- 为 `AI Provider` 定义 mock 或 fake 实现
- 测试业务层只验证输入组织和输出消费
- 单独保留手工验证脚本测试真实模型接入

## 7. 验收测试与 PRD 对齐

应至少建立 6 组验收场景，分别对应：

1. 报告项目与素材收集
2. 素材归集与用途标记
3. 复用定位
4. 主草稿生成
5. 主草稿改写
6. 最终报告归档

每组验收都应回答：

- 输入样例是什么
- 关键动作是什么
- 预期输出是什么
- 如何判断成功
- 哪些情况必须判失败

## 8. 测试目录建议

```text
tests/
  unit/
  integration/
  e2e/
samples/
  assets/
  expected/
```

补充说明：

- `tests/` 和 `samples/` 根目录可以先保留为空目录
- 样例文件的命名、映射和降级表达方式统一遵循 [样例约定](sample-conventions.md)

## 9. 执行建议

在项目初始化后，建议先让以下命令尽快可用：

```powershell
pytest
pytest tests/unit
pytest tests/integration
```

如果后续补充覆盖率统计，可增加：

```powershell
pytest --cov=draftcore --cov-report=term-missing
```

## 10. 测试策略验收标准

当以下问题都能直接回答时，可认为测试文档足以支撑开发：

- 哪些东西必须写单元测试
- 哪些流程需要集成测试
- 端到端验收怎么和 PRD 对齐
- 没有真实 AI 时怎么测试
- 解析器样例应该放哪里
