# ADR-001 技术栈与核心选型

## 1. 状态

已采纳。

## 2. 背景

DraftCore 当前处于第一版架构与开发准备阶段。项目边界已在 [实施说明.md](/C:/Codefield/DraftCore/实施说明.md) 和 [阶段5任务清单.md](/C:/Codefield/DraftCore/阶段5任务清单.md) 中明确：

- 单人使用
- 本地优先
- 报告工作流优先
- 多类型素材输入
- 第一版主输出为 Markdown
- 后续需要兼容 Word 工作流扩展

因此技术选型必须优先满足以下约束：

- 对 Windows 本地环境友好
- 对本地文件和多格式文档处理友好
- 可快速实现 CLI 工作流
- 具备低运维、低复杂度、易扩展特点

## 3. 决策

### 3.1 主语言：Python

选择 `Python` 作为第一版主实现语言。

原因：

- 适合本地文件处理、路径管理、SQLite 集成
- 文档处理生态成熟，便于后续支持 `docx`、`pptx`、`pdf`、图片等类型
- 适合快速构建 CLI 工具
- 适合后续接入 AI Provider 和文本处理流程
- 后续扩展 Word 工作流时成本更低

不选 `TypeScript/Node.js` 作为第一版主栈的原因：

- 虽然 CLI 生态成熟，但本项目更偏重文档解析和本地自动化
- 在多格式文档处理与脚本编排方面，Python 更直接

### 3.2 CLI 框架：Typer

选择 `Typer` 作为 CLI 框架。

原因：

- 命令组织清晰，适合工作流型 CLI
- 类型注解友好
- 参数说明和帮助信息成本低
- 适合逐步扩展子命令结构

### 3.3 数据库存储：SQLite

选择 `SQLite` 作为本地数据库。

原因：

- 适合单用户本地场景
- 零运维
- 足以支撑项目、素材、关系、草稿和归档查询
- 便于和本地文件系统形成简单清晰的双层存储结构

不选择纯文件化存储作为主方案的原因：

- 纯 JSON/YAML 在对象关系查询、筛选和后续演进上成本更高
- 项目与素材、草稿与素材、报告与来源的多对多关系更适合数据库表达

### 3.4 数据访问层：SQLModel

选择 `SQLModel` 作为数据库访问层。

原因：

- 基于 SQLAlchemy，能力够用
- 类型声明和模型定义相对统一
- 对第一版开发效率更友好

### 3.5 配置格式：TOML

选择 `TOML` 作为本地配置格式。

原因：

- 可读性较好
- 适合保存工作区、默认目录、AI Provider 配置等静态设置
- 与 Python 工具链习惯一致

### 3.6 文件解析策略：适配器模式

选择“按文件类型适配”的解析方式。

统一原则：

- 每种文件类型由独立解析器负责
- 解析器输出统一标准化结果
- 业务层不感知原始格式细节

第一版各类型处理策略如下：

| 文件类型 | 第一版策略 | 备注 |
| --- | --- | --- |
| `md` / `txt` | 直接读取文本、基础元数据 | 可直接参与检索和复用 |
| `docx` | 抽取段落文本、标题层级、基础元数据 | 不承诺高保真版式 |
| `pptx` | 抽取页标题、文本框文本、备注等最小文本信息 | 不做深层结构理解 |
| 图片 | 记录路径、名称、备注、用途 | OCR 后置 |
| `xlsx` | 预留接口，不列为第一版必做能力 | 后续扩展 |

### 3.7 输出渲染策略：导出器模式

选择“结构化草稿 + 导出器”模式。

第一版实现：

- `MarkdownExporter`

后续预留：

- `WordExporter`

原因：

- Markdown 是第一版最轻量、最稳定的输出目标
- 结构化草稿能避免业务逻辑和 Markdown 语法强耦合
- 后续增加 Word 输出时，只需新增导出器，而不是重构草稿生成链路

### 3.8 检索策略：关键词与元数据优先

第一版选择关键词、字段和抽取文本检索，不引入向量检索。

原因：

- 可解释性强
- 实现成本低
- 足以支撑 MVP 的复用定位任务
- 避免提前引入 embedding、向量库存储和额外运维复杂度

### 3.9 AI 接入策略：Provider 适配层

第一版选择通过统一 `AI Provider` 接口接入云端模型能力。

原则：

- AI 是增强能力，不是基础闭环前置
- 业务层不直接依赖具体厂商 SDK
- 接口至少覆盖 `generate_draft` 和 `rewrite_text`
- 输入使用标准化素材片段，而非绑定原始文件格式

## 4. 后置能力

以下能力明确不纳入第一版实现，但必须在架构上预留边界：

- Word 导出
- 图片 OCR
- PPT 深度内容结构化理解
- 高保真文档版式还原
- 向量检索与语义检索基础设施

## 5. 影响

### 5.1 正面影响

- 技术栈简单，便于个人项目推进
- 与多格式文档处理需求匹配
- 能尽快支撑阶段 5 到阶段 6 的过渡
- 为 Word 工作流扩展保留了稳定接口

### 5.2 代价与限制

- 第一版在文档保真度上刻意保守
- 图片无法直接参与正文语义复用
- `pptx` 只能提供最小文本抽取，不支持复杂内容理解
- 后续如要加入语义检索，需要新增独立能力层

## 6. 决策完成标准

当以下问题不再需要重复讨论时，可认为本 ADR 生效：

- 主语言是什么
- CLI 用什么框架
- 数据库存什么
- 用什么 ORM
- 配置文件用什么格式
- 多文件类型通过什么模式接入
- 第一版输出为什么是 Markdown
- 后续 Word 为什么不需要推翻当前设计
